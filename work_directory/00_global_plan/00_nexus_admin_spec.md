# nexus_admin — спека проекта (первая очередь)

Дата: 2026-06-10 · Источник: `00_nexus_admin_init.md` (интервью + решения 1–13)
Статус: к декомпозиции (фаза plan)

---

## 1. Цель

Личный кабинет управленца для системы nexus: единое место, где видно **план-факт по всем
дев-проектам организации** (часы, даты, прогресс, токены, ≈API-стоимость) с drill-down от портфеля
проектов до отдельной AI-модели в задаче. Стратегически — будущая замена Plane (контроль задач и
проектов переезжает в кабинет).

## 2. Результаты (deliverables первой очереди)

1. **Каркас кабинета:** Next.js 16 + TypeScript + Tailwind 4 + shadcn/ui; layout с sidebar
   (Проекты · Сотрудники · База знаний · Отдел продаж — последние три как заглушки);
   дизайн по референсу Logip (Dribbble), компоненты shadcn/ui (+21st.dev при необходимости).
2. **Auth и роли:** вход email+пароль, только по инвайтам; роли **Owner / Admin / Employee / Client**;
   страница управления пользователями (добавить/удалить/сменить роль).
3. **Слой данных план-факт:**
   - схема кабинета в Supabase Postgres (пользователи, роли, проекты-метаданные, план-кэш);
   - sync-job (Node, Windows Task Scheduler): реестр `~/.wgp/projects.json` → каноны
     `00_<slug>_plan.json` → Postgres; описания проектов из орг-памяти (cerebro);
   - факт read-only из таблиц timechecker (`daily_task_time`, `daily_agent_usage`, `agent_session`).
   - **Plane в продукте не используется.**
4. **Раздел «Проекты»:**
   - обзор: переключатель активные/завершённые/все, карточки с описанием и % выполнения;
   - **Гант**: горизонталь — даты, вертикаль — проекты; полоса = период проекта, заливка = % выполнено,
     деления — эпохи, hover эпохи → тултип (описание, факт-часы, токены, ≈API-стоимость);
   - **drill-down проекта**: эпохи → спринты → задачи (даты, план/факт-часы, статус, токены,
     ≈стоимость; у задач — исполнитель) → **модели AI** (1 строка = 1 модель: время, токены, ≈стоимость);
   - role-based видимость: Employee видит только свои проекты (Client — заготовка).
5. **Деплой:** Docker на Linux VPS (Ubuntu 24.04, 2 vCPU/4 GB), доступ по IP, заготовка под домен/HTTPS.
6. **Документация:** README (запуск, синк, доступы), ASCII-схемы страниц в спеках.

## 3. Критерии приёмки (первая очередь)

- Управленец входит по своему аккаунту (Owner) и видит обзор: все проекты реестра
  (nexus, timechecker, amo_looker) с описаниями, фильтром и актуальным % (= `done_h/total_h` канона).
- Гант отображает реальные плановые даты и эпохи всех активных проектов; тултип эпохи показывает
  фактические часы, токены и ≈API-стоимость, сходящиеся с отчётами timechecker.
- Drill-down любого проекта раскрывается до уровня задач (с исполнителем) и далее до моделей AI;
  цифры по токенам/стоимости на уровне задачи сходятся с timechecker (допуск — округление).
- Owner/Admin создаёт инвайт, новый пользователь устанавливает пароль и входит; Employee видит
  только свои проекты; удаление пользователя и смена роли работают.
- Приложение работает на Linux VPS по IP; синк обновляет данные по расписанию без ручных действий.
- Каждая страница реализована только после согласованной ASCII-схемы.

## 4. Ограничения и решения

- Стек: **Next.js + shadcn/ui** (вариант B утверждён); дизайн-референс Logip; язык RU; стоимость USD «≈ API-эквивалент».
- БД: **одна Supabase Postgres**, две зоны: схема кабинета (read/write) + схема timechecker (read-only).
- «Задача × модель»: агрегирование `agent_session` по task × model (решение управленца, timechecker
  не дорабатываем); стоимость = токены × ставки `~/.wgp/pricing.json`; ограничение: модель одна на
  сессию, мульти-модельные сессии приписываются доминирующей модели.
- Sync-job живёт на Windows-сервере (рядом с канонами и реестром), пишет в облачный Postgres;
  кабинет читает только Postgres.
- Git: репо `gram2Claude/nexus_admin`, target `master`, dev-ветка `oleg`.
- Правило «завершённого» проекта (для фильтра обзора): **все задачи канона в Done** — предложено
  по умолчанию, подтвердить у управленца (t21).

## 6. Риски и зависимости

- **VPS ещё не выделен** (есть готовность выделить) — эпоха деплоя зависит от предоставления сервера;
  до этого работаем локально. ETA сервера уточнить к началу последней эпохи.
- Маппинг задач кабинета ↔ timechecker идёт через PLANE-ID (`task` в timechecker зеркалится из Plane).
  При будущем отказе от Plane источник `task` в timechecker придётся переключать — вне скоупа первой
  очереди, но архитектуру маппинга делаем через нейтральный ключ (readable id), не через plane-uuid.
- Описания проектов в орг-памяти есть не для всех проектов (cerebro покрывает timechecker, план-факт,
  nexus-лендинг) — для остальных описание заполняется в кабинете вручную (поле редактируемо).
- Supabase: кабинет получает **отдельные креды** с ограничением прав (timechecker-схема read-only) —
  не использовать общий `supabase_db_url` владельца в продакшен-приложении.
- Публичный репозиторий + код кабинета: секреты только в env на сервере, не в репо.
- **Bleeding-edge стек:** Next.js 16 + Tailwind 4 + Auth.js credentials (edge runtime в middleware,
  bcrypt) — известные грабли совместимости; fallback — Supabase Auth (t06).
- **Дрейф формата канонов:** три реальных `plan.json` писались разными версиями WGP — sync-job (t13)
  должен переживать различия, сверка против всех трёх обязательна.
- **Две методики стоимости:** precomputed `cost_usd` дневного агрегата vs токены×pricing из сессий
  могут расходиться; допуск ±1% задан заранее (t18), сверка — t19.
- **Качество сессионных данных:** NULL task_id / NULL ended_at / перекрывающиеся сессии в
  `agent_session` — уровень «модель AI» строится на них (t18); затраты без задачи — бакет
  «нераспределённое» (t26).
- **Свежесть факта = лаг репликации timechecker→Supabase:** кабинет ею не управляет; при сверке
  фиксируется момент репликации (t19), в UI — метка свежести + warning (t15).
- **Windows-сервер с sync-job — единая точка отказа:** при его недоступности план тихо устаревает;
  warning-баннер при свежести > 12 ч (t15).
- **Общая Supabase БД с timechecker:** roll-up запросы кабинета бьют в прод-БД учёта — пулинг
  соединений + кэширование агрегатов (t17), лимиты тарифа Supabase проверить при деплое.
