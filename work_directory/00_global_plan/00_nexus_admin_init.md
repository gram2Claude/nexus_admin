## 1. Сырое описание от управленца (исходный запрос)

Полноценный личный кабинет для управленца в системе nexus.

**Разделы (первая очередь):**
- Проекты ← **начинаем с этого**
- Сотрудники
- База знаний
- Отдел продаж
- (новые разделы будут добавляться по мере продвижения)

**Раздел «Проекты»:**
- Переключатель: активные / завершённые / все.
- У проекта — краткое описание (что он из себя представляет).
- Подраздел «Общий обзор активных проектов»:
  - что в работе, % выполнения;
  - диаграмма Ганта: горизонталь — даты, вертикаль — проекты;
  - горизонтальная полоса = период проекта; закрашенная часть = выполнено, штриховка/белое = осталось;
  - рядом подпись с % выполнено / осталось;
  - полоса делится вертикальными линиями на эпохи;
  - hover на блок эпохи → тултип: краткое описание эпохи, фактические часы, токены, ≈API-эквивалент стоимости.
- Переход в конкретный проект — drill-down от общего к частному (таблица-матрица или аккордеон):
  - **Уровень эпох:** какие эпохи запланированы, описание, даты начала/окончания, плановые часы,
    статус (завершена / в работе / в ожидании), фактические часы (если есть), токены + ≈API-стоимость.
  - **Уровень спринтов:** то же в разрезе спринтов.
  - **Уровень задач:** то же в разрезе задач + исполнитель.
  - **Уровень AI-моделей (клик по задаче):** 1 строка = 1 модель;
    время использования модели, токены, ≈API-эквивалент стоимости.

**Пользователи и роли:**
- добавление/удаление пользователей, управление ролями;
- 4 роли: **Owner, Admin, Employer, Client**.

**Процессные требования:**
- разделы разрабатываются последовательно, описание остальных — по мере продвижения;
- **для каждой страницы сначала ASCII-схема → согласование → только потом визуализация** (конвенция зафиксирована);
- компоненты — **shadcn/ui** (https://ui.shadcn.com/);
- при необходимости управленец выделит отдельный сервер.

**Запрошена рекомендация по пути:**
- a) Laravel + Filament;
- b) самописный кабинет план-факт, связанный с Plane (в перспективе — уход от Plane, контроль задач/проектов переносится в кабинет);
- c) учесть, что есть система timechecker (учёт времени и токенов).

## 2. Что изучено (контекст организации)

| Система | Что даёт кабинету | Где данные |
|---|---|---|
| **workflow_global_plan** (план-факт) | ПЛАН: иерархия Global→Эпохи→Спринты→Задачи, часы (`estimate_h`, `ai_estimate_h`), даты, статусы, Plane-ID | канон `work_directory/00_global_plan/00_<slug>_plan.json` в каждом проекте; реестр `~/.wgp/projects.json` (сейчас: nexus, timechecker, amo_looker) |
| **Plane** | живое зеркало статусов задач (In Progress / Done), ручные задачи | REST API (`plane_api_key` в `~/.wgp/secrets.json`) |
| **timechecker** | ФАКТ: время по задачам (`daily_task_time`), токены/кэш/стоимость по агентам (`daily_agent_usage`: день × задача × source claude/codex), сессии агентов (`agent_session`, поле `model`), сотрудники, простои | local-first SQLite → **Supabase Postgres** (копия-архив; `supabase_db_url` уже в `~/.wgp/secrets.json`) |
| **nexus** (07_nexus) | стек-ориентир: **Next.js 16 + React 19 + Tailwind 4 + TypeScript** | репо gram2Claude/nexus |
| **Репо кабинета** | уже создан: gram2Claude/nexus_admin, ветка `main`, каркас work_directory | текущая папка 16_nexus_admin |

**⚠ Найденный технический нюанс (важно для нижнего уровня drill-down):**
`daily_agent_usage` агрегируется по **агенту** (claude/codex), а НЕ по модели. Модель хранится на
уровне сессии (`agent_session.model`, одна на сессию). Для уровня «задача × модель» есть два пути:
1. агрегировать `agent_session` по task × model (время = сумма длительностей сессий) — без доработок timechecker, но грубее;
2. **доработать timechecker**: добавить разрез model в `daily_agent_usage` (или новый агрегат) — точно и дёшево, timechecker наш.
**Рекомендация: путь 2** (небольшая доработка timechecker войдёт отдельной задачей в план кабинета).

## 3. Рекомендация по стеку — 3 варианта

### Вариант A (минимальный по сроку старта): Laravel + Filament
Готовая админ-панель: CRUD, таблицы, формы, роли — из коробки.
**Против (перевешивает):**
- **Несовместим с shadcn/ui.** Filament = PHP/Livewire/Blade, shadcn = React. Желание «компоненты на shadcn» в Filament нереализуемо — будет UI Filament.
- PHP — **новый стек для организации** (всё существующее: Python + Next.js/TypeScript). Третий стек = расходы на поддержку навсегда.
- Главные экраны кабинета — НЕ CRUD: Гант, drill-down план-факт, тултипы с токенами. В Filament это кастомные виджеты против течения фреймворка.
- «Уйти от Plane» = со временем строить свой таск-менеджер — в чужой админ-парадигме это дороже.
- Интеграции (Supabase Postgres + Plane REST + канон JSON) пришлось бы писать заново на PHP.

### Вариант B (рекомендуемый): самописный кабинет — Next.js + shadcn/ui
- **Стек = стек nexus** (Next.js 16 / React 19 / Tailwind 4 / TS) — одна экосистема, кабинет «в системе nexus» буквально.
- **shadcn/ui — нативно** (он для React/Next.js и создан), графики — Recharts (официальные shadcn charts), Гант — кастомный SVG/div-компонент (для нашего формата «полоса с эпохами и заливкой %» готовых библиотек всё равно нет — рисуем сами, это управляемо).
- **Данные уже лежат в Postgres (Supabase)** — timechecker туда реплицирует; кабинет читает их напрямую (read-only), а свои сущности (пользователи, роли, описания проектов, кэш плана) держит в отдельной схеме той же БД.
- ПЛАН приходит синком из канонов `00_<slug>_plan.json` + статусы из Plane REST.
- **Путь ухода от Plane встроен в архитектуру:** этап 1 — кабинет читает Plane (зеркало); этап 2 — сущности задач переезжают в свою БД, Plane отключается. С Filament такой эволюции не получится без переписывания.
- Auth + 4 роли: Auth.js (NextAuth) или Supabase Auth + таблица ролей; инвайты от Owner/Admin.

### Вариант C (латеральный): Next.js + Refine (refine.dev) + shadcn
Refine — React-каркас для админок (готовые CRUD-хуки, auth/data-провайдеры, включая Supabase).
Ускоряет CRUD-часть (пользователи/роли), но добавляет слой абстракции, а CRUD в нашем кабинете —
малая доля; главные экраны (Гант, drill-down) Refine не ускоряет. **Не рекомендую** — выгода не
окупает зависимость.

### ИТОГ-РЕКОМЕНДАЦИЯ: **Вариант B.**
Laravel/Filament противоречит трём заявленным требованиям сразу (shadcn, связь с имеющимися данными,
перспектива «свой таск-менеджер вместо Plane»). Вариант B использует уже существующую инфраструктуру
(Supabase, Plane API, каноны план-факта) и стек организации.
ОТВЕТ (утверждаем вариант B?): да, утверждаем Вариант B

**Архитектура (эскиз, вариант B):**
```
┌──────────────────────── nexus_admin (Next.js + shadcn/ui) ───────────────────────┐
│  UI: Проекты (Гант, drill-down) · Пользователи/Роли · (позже: Сотрудники, БЗ, …)│
```

## 5. Решения (зафиксированы 2026-06-10 по ответам управленца)

1. **Стек: вариант B — самописный кабинет на Next.js + shadcn/ui.** Утверждено.
2. **Развёртывание:** Linux VPS в облаке, доступ сначала по IP сервера, домен/HTTPS — позже.
   **Требования к железу (от Claude):** БД в облаке Supabase уже есть, на сервере крутится только
   Next.js-приложение → достаточно **2 vCPU, 4 GB RAM, 40 GB SSD, Ubuntu 24.04 LTS + Docker**
   (минимум 2 GB RAM; типичный VPS за $5–10/мес). Node.js 22 LTS внутри контейнера.
3. **Кабинет — отдельное приложение** (не раздел сайта nexus).
4. **Роли:** Owner — всё + управление админами; Admin — пользователи и контент, не трогает Owner;
   **Employee** (уточнено имя роли — «сотрудник») — видит только проекты, в которых участвует;
   Client — про запас, права пропишем отдельно позже.
5. **% выполнения** — по часам: `done_h / total_h` из канона план-факта.
6. **Plane в кабинете НЕ участвует вовсе** (курс на отказ от него): план читается из канонов
   `00_<slug>_plan.json` (реестр `~/.wgp/projects.json`), краткие описания проектов — из
   долгосрочной памяти организации (cerebro) + редактируемое поле в кабинете.
7. **Уровень «задача × модель AI»:** агрегируем `agent_session` (timechecker) по задача × модель;
   стоимость считаем из токенов × ставки; timechecker НЕ дорабатываем (решение управленца, ограничения зафиксированы в В6).
8. **Вход:** только инвайты от Owner/Admin, email + пароль, саморегистрации нет.
9. **Язык/валюта:** RU-интерфейс, стоимость в USD «≈ API-эквивалент».
10. **Git:** целевая ветка переименована `main` → **`master`** (на GitHub и локально, выполнено);
    ветка разработчика — `oleg`; `start_date` = **2026-06-10** (управленец ответил «да» без даты —
    принято «сегодня», подтвердить на чекпоинте); жёсткого дедлайна нет.
11. **Скоуп первой очереди подтверждён:** раздел «Проекты» (обзор + Гант + drill-down до моделей) +
    каркас кабинета (auth, роли, пользователи) + заглушки разделов «Сотрудники», «База знаний», «Отдел продаж».
