# Двойное ревью кода спринта 2.2 (NEXADM-9, 10) — итоги триажа

Дата: 2026-06-11. Ревьюеры: Claude-субагент + codex (high reasoning), независимо.
Объект: коммит `7d3c1c9` (инвайт-флоу + страница «Пользователи»). Совпадение находок ревьюеров
высокое: оба нашли bcrypt-DoS, deadlock порядка локов, disabled-тупик, токен в URL.

## Применено (15 правок)

| # | Находка | Кто | Правка |
|---|---|---|---|
| 1 | **CPU-DoS**: bcrypt (≈300 мс CPU) выполнялся ДО проверки токена на публичном эндпоинте | оба (major/P1) | bcrypt только после подтверждения существования валидного токена + rate-limit per-IP на set-password (5/15 мин) |
| 2 | **consumeInvite без rowCount**: при users-строке не в invited токен гасился, UI говорил «пароль сохранён», хотя нет | субагент major | `rowCount === 0` → ROLLBACK + null |
| 3 | **disabled-тупик**: перевыпуск инвайта disabled-пользователю не возвращал статус invited — активация невозможна никогда | оба | createInvite ставит `status='invited'` + сбрасывает password_hash |
| 4 | **Deadlock**: createInvite лочил users→invites, consumeInvite — invites→users | codex P1 / субагент | единый порядок users→invites везде; consumeInvite перепроверяет токен под локом |
| 5 | deleteUser: SELECT + 2 DELETE без транзакции — гонка с инвайтом давала «осиротевший» валидный токен | субагент | транзакция + FOR UPDATE |
| 6 | Обход guard'а перевыпуска: инвайт на email pending-admin'а через диалог тихо понижал его роль | субагент | RBAC-проверки существующей строки перенесены в createInvite под лок (один источник для invite/reinvite) |
| 7 | Сырые PG-ошибки уходили в UI (`e.message`) | субагент | класс InviteError: в UI только бизнес-тексты |
| 8 | Рассинхрон UI/сервер: UI запрещал admin→admin, сервер разрешал | субагент | политика закреплена в `canModifyUser`: admin трогает только employee/client (privilege containment) |
| 9 | Пароль без верхней границы (bcrypt усекает после 72 байт + работа хэшера на мегабайтах) | субагент | лимит 72 в set-password (server+form) и в authorize |
| 10 | Токен в GET-URL: логи/referrer | оба (minor/P2) | `referrer: no-referrer` на странице; в t31 — не логировать query этого пути; полный POST-handoff отклонён (ниже) |
| 11 | Select роли неуправляемый — после ошибки висел отклонённый вариант | субагент | key={id-role} → ремоунт при изменении |
| 12 | Ссылка прошлого инвайта «залипала» в диалоге — риск отдать чужую ссылку | субагент | ремоунт формы при каждом открытии диалога |
| 13 | Погасшие reinvite-ссылки оставались на экране после удаления пользователя | субагент | очистка стейта при удалении |
| 14 | clipboard API вне secure context молча падал | субагент | try/catch + fallback на выделение текста |
| 15 | (следствие №8) UI: admin не видит действий над другим admin | — | editable-условие согласовано с rbac |

## Отклонено

| Находка | Кто | Почему |
|---|---|---|
| POST-handoff/fragment вместо токена в query | codex P2 (полный фикс) | переусложнение для первой очереди: токен одноразовый, TTL 7 дней, no-referrer стоит, ссылку передаёт сам Owner/Admin; запрет логирования query — в t31 |

## Чисто (проверено обоими)

RBAC во всех server actions (requireManager + canModifyUser + assignAdmin, прямой вызов мимо UI не проходит);
токен 256 бит CSPRNG, в БД только SHA-256, не логируется; одноразовость под FOR UPDATE (двойная активация
отбивается); инвайт на активный/owner email — отказ; в клиент уходят только безопасные поля; SQL
параметризован; CSRF закрыт; NEXT_REDIRECT пробрасывается.
