# Спека спринта 3.1 — План: схема и синк канонов (NEXADM-12…15)

Статус: **В РАБОТЕ** (режим «без остановок», утверждён управленцем 2026-06-11).
Эпоха backend: новых полноценных страниц нет; утверждение деталей — на СТОПе перед merge.

## 1. Схема план-данных (t12, миграция 003)

Таблицы в схеме `nexus_admin`: `projects` / `epochs` / `sprints` / `tasks` — зеркала канонов
(ключ `(project_id, ext_id)`, у задач плюс `readable_id` = NEXADM-N — нейтральный ключ к
timechecker); `pricing` (модель → ставки in/out/cache-write/cache-read $/Mtok); `project_members`
(membership для t28); `sync_meta` (метка свежести). Tombstone: исчезнувшее из канона помечается
`archived`, не удаляется. Roll-up `done_h` считается на синке. Статус проекта: `completed` =
все задачи done (правило утверждено в чекпоинте плана).

Дрейф канонов (проверено по трём реальным): у nexus/timechecker нет `description` у задач и эпох,
поля `started_at/merge_sha/dev_branch` плавают — синк терпим к отсутствию любых необязательных полей.

## 2. Sync-job (t13, scripts/sync/sync-plan.mjs)

Запуск на Windows-сервере (рядом с канонами): реестр `~/.wgp/projects.json` (enabled) →
канон каждого проекта → идемпотентный upsert иерархии → tombstones → roll-up'ы → статус проекта;
`~/.wgp/pricing.json` → upsert `pricing` (ставки нужны кабинету на VPS); `sync_meta.last_sync_at`.
Повторный прогон без изменений = no-op. Описания проектов синк НЕ трогает (см. п.3).

## 3. Описания проектов (t14)

Поле `projects.description` живёт только в кабинете: первичное наполнение — из орг-памяти
(cerebro) для известных проектов, далее правится Owner/Admin и никогда не перетирается синком.
Редактирование — на странице /projects (черновой список, см. п.4).

## 5. Планировщик и свежесть (t15)

`scripts/sync/register-sync-task.ps1` → 3 задачи Task Scheduler (08:05/14:05/17:05 МСК — после
WGP-синка), лог в файл `scripts/sync/sync.log`. Header кабинета: «обновлено N мин назад» из
