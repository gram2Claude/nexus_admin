# Спека спринта 3.2 — Факт из timechecker (NEXADM-16…19)

Статус: **В РАБОТЕ** (режим «без остановок»). Backend без новых страниц.

## 1. Ключевое открытие данных и решение по «задача × модель» (t18)

В timechecker `agent_session.task_id` **всегда NULL** (атрибуция к задачам у них происходит
в дневных агрегатах, не на сессиях). Прямой group by сессий по задача×модель невозможен.

**Решение (развитие согласованного «агрегировать agent_session», В6):** дневные числа задачи
(`daily_agent_usage` — точные токены/кэш/стоимость; `daily_task_time` — минуты) распределяются
по моделям **пропорционально токен-долям моделей в сессиях того же сотрудника и дня**.
Свойства: суммы по задаче сходятся с timechecker **по построению** (сверка показала 0.00%
расхождения по всем задачам — допуск ±1% выполнен с запасом); модельная разбивка —
приближение в рамках дня.

**Отклонение от плановой формулировки t26:** бакет «нераспределённое» получается уровнем
**портфеля**, не проекта — у строк без задачи в данных timechecker нет и проекта
(claude-сессии не несут project_id). UI эпохи 5 покажет его отдельной строкой портфеля.

## 2. Архитектура доступа (t16)

- **View-слой** (миграция 004): `v_task_fact` (часы/токены/стоимость/исполнитель по задаче),
  `v_task_model_fact` (задача × модель). Доступ через owner-цепочку прав.
- **Роль приложения `nexus_admin_app`**: USAGE+SELECT на схему кабинета, INSERT/UPDATE/DELETE
  только на users/invites/project_members, UPDATE(description) на projects;
  **на public.* (timechecker) грантов НЕТ вовсе** — писать туда приложение физически не может,
  читать — только через view. Границы проверены реальным логином роли: 3/3 запрещённых
  операций отклонено. Скрипты (синк/миграции) остаются на админском DSN
  (`DATABASE_URL`), приложение — на `DATABASE_URL_APP`.
- Маппинг план↔факт: `tasks.readable_id` (NEXADM-N) = `public.task.identifier` (+ резерв
  `canon_task_id`); покрытие проверено на живых данных.

## 3. Запросный слой (t17, src/server/fact.ts)

`taskFactsByProject`, `levelFactRollup` (спринт/эпоха), `modelFactsByTask`,
`unallocatedModelFacts` — поверх view, server-only. Производительность: полный
`v_task_model_fact` — **34 мс** (p95 < 500 мс выполнен с 15-кратным запасом).

## 4. QA-сверка (t19)

Отчёт: work_directory/04_reviews/06_sprint3_2_qa.md. Главные числа: расхождение сумм
моделей с daily_agent_usage = 0.00% по всем задачам с данными; исполнитель определяется

фиксируется меткой свежести синка.
